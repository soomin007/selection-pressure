# 알려진 함정 / 반복 금지 (Known Issues)

> 버그·설계 함정·작업 실수를 발견하면 여기에 "증상 → 원인 → 재발 방지책"으로 남긴다.
> 게임 버그뿐 아니라 프로세스 실수(도구 오용, 커밋 누락 등)도 포함.
> 세션 시작 루틴에서 이 파일을 먼저 읽어 같은 실수를 예방한다.

### sim 효과를 넣으면 관련 시각(부채꼴·도감·범례)도 반드시 함께 갱신 (일부만 고치면 어긋남)
- 증상: 수풀 시야 감소를 behavior(sim vision)에만 넣고, 화면 시야 부채꼴(worldView playerVision)과
  대백과 시야 그림에는 반영을 빼먹음. → 수풀 안에서 실제론 시야가 줄었는데 부채꼴은 그대로라 "수풀서
  잘 쫓아다닌다"로 보이고, 도감 그림도 부정확. 사용자 질책("수정할 때 빼먹지 말라").
- 원인: 한 효과(지형×형질)가 여러 표면(sim 계산·부채꼴 렌더·도감 SVG/설명·범례)에 걸쳐 있는데 sim만
  고침. 시야 부채꼴은 종 게놈 "고정값"이라 개체 위치(수풀)와 무관하게 그려지고 있었다.
- 재발 방지책: **새 sim 효과를 넣거나 형질/지형 계산을 바꿀 땐 그 효과가 보이는 모든 표면을 체크리스트로
  훑는다** — ① sim 계산 ② 그 값을 그리는 렌더(부채꼴·오라·틴트 등, "고정값"으로 계산하던 곳은 개체별로)
  ③ 대백과(SVG 그림 + 수치 + 설명) ④ 범례/HUD. 렌더가 sim 과 같은 함수를 쓰게 하면(behavior 의
  grassVisionFactor 를 worldView 도 import) 어긋남이 원천 차단된다(시각=로직 1:1의 실천).

### 이동체를 지형에 새로 구속시키면 길찾기도 함께 줘야 한다 (안 그러면 코앞에서 영영 못 잡는다)
- 증상: 땅 보스가 물·산을 못 넘게 막았더니(층위 도입), 보스가 먹잇감을 20~30px 앞에 두고도 끝까지 못
  잡았다. 화면상 계속 "움직이고는 있어서"(600틱 내내 이동) 멈춘 걸로도 안 보였다. 테스트 1건("추격자는
  보스 사망으로 집계된다")이 0 kill 로 실패해서야 드러났다.
- 원인: 이동을 "목표 방향으로 직진 + 축 분리 슬라이딩"으로만 두면, 물가·산자락에서 벽을 따라 좌우로
  미끄러지기만 하고 목표에 못 간다(반응형 조향의 local minima). 이미 개체(behavior)에서 겪고 격자 BFS
  (`navTo`)로 해결한 문제인데, **보스는 원래 지형을 무시하고 다녔기 때문에** 그 교훈이 안 옮겨져 있었다.
- 해결: `bossNavTo`(boss.ts) — 개체의 navTo 와 같은 구조(직선 보이면 직진 / 막히면 findPath 캐시 + funnel).
- 재발 방지책: **"이제부터 이 이동체는 지형에 막힌다"를 도입할 땐 같은 커밋에서 길찾기를 붙인다.** 통행
  판정(isPassable)만 추가하는 것은 절반짜리다. 진단 팁: "움직이는가"가 아니라 **"목표까지의 거리가 줄어드는가"**
  를 재라 — 벽을 따라 미끄러지는 것도 "움직임"이라, 이동량만 보면 정상으로 보인다(avoidWalls 때와 같은 맹점).

### 형질 카운터를 "절대 개체수"로 재면 비생존 빌드를 위협 탓으로 오독한다
- 증상: 새 보스(큰수리)의 시야 카운터를 재는데 "시야10 → 개체수 0.2 / 시야90 → 6.5" 라 카운터가 잘 듣는
  줄 알았다. 그런데 시야90 이 시야50(11.3)보다 **낮았다** — 카운터가 뒤집힌 것처럼 보였다.
- 원인: **보스 없는 기준선을 안 쟀다.** 실제로는 시야10 이 보스가 없어도 개체수 0.3(0/8 통과)인 비생존
  빌드였고(먹이를 못 찾아 굶는다), 시야90 도 기준선이 시야50 보다 낮았다(12.5 vs 15.3). 즉 그 숫자들은
  보스와 거의 무관했다. 게다가 도망 반경(visionFlee)을 90 까지 키우면 시야 큰 종이 내내 달아나느라 못 먹고
  굶는다(공황 아사) → 카운터를 세게 만들수록 오히려 뒤집히는 구간이 있다(60 이 최적).
- 재발 방지책: 카운터 형질을 잴 땐 **반드시 "그 위협이 없을 때"의 기준선과 함께 재고, 차이(위협의 영향)로
  판단한다.** 절대 개체수는 그 형질 자체의 생존력에 지배당한다. 그리고 회피 파라미터(도망 반경 등)는 키울수록
  좋은 게 아니다 — 채집 시간을 갉아먹는 지점이 있으니 값을 쓸어 보고 최적점을 찾을 것.

### GitHub Pages 첫 배포 시 configure-pages 권한 에러
- 증상: `actions/configure-pages@v5` 가 "Resource not accessible by integration" 로 실패,
  Pages 사이트 생성 안 됨.
- 원인: 워크플로 기본 `GITHUB_TOKEN` 권한으로는 Pages 사이트를 처음 만들지 못함.
- 재발 방지책: 레포 생성 직후 사용자 토큰으로 Pages 를 먼저 활성화한다 →
  `gh api --method POST repos/<owner>/<repo>/pages -f build_type=workflow`
  (Git Bash 는 leading slash 를 경로로 오인하니 슬래시 없이). 그 후 워크플로 재실행.

### GitHub Pages 배포가 "Deployment failed, try again later"로 계속 실패(배포 stuck) → 옛 버전만 뜸
- 증상: 빌드(artifact 생성)는 성공하는데 `deploy-pages` 단계가 "Deployment failed, try again later"로
  실패. 여러 커밋 동안 반복 → 최신 코드가 사이트에 안 올라가고 옛 버전만 뜬다(폰·시크릿 창 모두). "새
  기능이 화면에 안 보인다"를 코드/캐시 문제로 오해하기 쉽다(실제론 배포가 안 된 것).
- 원인: 이전 Pages 배포가 GitHub 백엔드에서 stuck. 특히 짧은 간격으로 여러 번 push 하면(concurrency
  `cancel-in-progress` 로 cancelled 가 쌓임) 배포 큐가 엉켜 이후 배포가 전부 거부된다.
- 재발 방지책: **"안 보인다"는 피드백이 오면 코드/캐시보다 `gh run list` 로 최근 Deploy 성공 여부부터
  본다.** 실패면 `gh run rerun <run-id> --failed`(deploy job만 재실행)로 대개 풀린다. 여러 커밋을 몰아
  push 하지 말고 하나씩 배포를 확인. 지속되면 `gh api repos/<o>/<r>/pages` 로 `build_type=workflow` 확인.

### sim 행동을 바꾸면 개체 수 스케일이 변해 절대 기준 밸런스 테스트가 깨진다
- 증상: 이동 로직(관성/목표 고정)을 매끄럽게 했더니 채집 효율이 올라 개체 수가 전반 상승,
  한파 대멸종 필터의 절대 통과기준(저대사<10) 테스트가 실패(저대사가 14로 생존).
- 원인: 게임 밸런스가 "절대 개체 수 기준"으로 잡혀 있어, 행동 효율이 바뀌면 분포가 통째로 이동.
- 재발 방지책: sim 행동을 바꾸면 무작정 상수만 만지지 말고 **임시 프로브 테스트로 분포부터 측정**
  (forage/poison/cold/heat/chaser/sanity 한 번에 console.log) 후 타깃 튠. 평상시에 영향 없는
  지점(예: 대멸종 전용 배율)을 골라 고치면 sanity/Phase3 회귀를 피한다.

### 종 다양성: 단일 먹이 자원에서 경쟁 배제로 금방 1~2종만 남음
- 증상: 야생종을 늘려도(3→6종) 수천 틱이면 1~2종(보통 내 종)만 남음. 시드 편차 큼.
- 원인: 모든 초식종이 같은 먹이 한 종류를 두고 경쟁 → 경쟁 배제 원리상 한둘만 살아남음.
  36마리로 시작하는 내 종이 동등 채집력으로 수적 지배까지 함.
- 완화책(현재): 생존 전략 분리(다산/저대사 장수/잡식/포식) + 야생 영역(클러스터) 스폰으로
  초·중반 공존을 늘림. 후반 붕괴는 못 막음.
- 근본 해결(적용 478e1d1): (1) 먹이 종류 분할 — food.kind 3종, 야생 초식종마다 다른 종류 전문이라
  서로 경쟁 안 함(전문종 공존). (2) 야생 주기적 이주 — 적은 야생종을 10초마다 소수 보충(멸종 바닥).
  결과: 4시드 전부 t750~t4000 내내 7종 전부 유지. 교훈: 단일 자원 경쟁 배제는 자원을 쪼개야 풀린다.

### 월드 스케일(areaScale) 도입 시 절대 상수를 빠뜨리면 그 축에서 분포가 붕괴
- 증상: 맵 3배(areaScale 9) 후 내 종은 늘고 야생만 단조 급감(t2000 야생 4%). 작은 월드(scale 1)는
  야생이 회복 진동(46%)하는데 큰 월드만 회복 못 함.
- 원인: areaScale 로 개체·먹이·cap·통과기준은 면적 비례시켰지만 **두 절대 상수를 빠뜨림** —
  ① 야생 보금자리 spread(72px): 큰 맵에서 좁은 점에 과밀 → 국소 먹이 소진 후 집단 아사.
  ② 이주 floor 4·batch 3: 큰 맵 종당 ~90 대비 4%라 보충이 무의미 → 줄어도 회복 못 함.
- 해결(4f0a45a): spread = 72·sqrt(areaScale)(길이라 제곱근), 이주 floor·batch = ×areaScale.
  큰 월드 야생 t2000 4%→48%(내 종과 균형).
- 재발 방지책: **월드 크기/스케일을 바꿀 땐 "모든" 절대 상수를 면적 기준으로 점검하라** — 개체수·
  먹이수·cap·통과기준(면적 비례) / 보금자리·이동거리 같은 "길이"(제곱근 비례) / 이주 바닥·보충량.
  하나만 절대값으로 남으면 그 축에서만 밀도가 어긋나 분포가 한쪽으로 쏠린다. 작은 vs 큰 월드 같은
  지표를 A/B 로 재서 "큰 월드만의 이상"을 가르는 게 진단의 핵심(원래 그런 것 vs 스케일 버그).

### 렌더 보간 없으면 sim 을 부드럽게 해도 화면은 드득거린다
- 증상: 행동 로직을 매끄럽게 고쳤는데도 생물이 잔떨림(특히 정지 화면/로비에서).
- 원인: sim 은 30스텝/초, 화면은 60fps. 보간 없이 sim 위치를 그대로 찍으면 두 프레임마다 점프.
- 재발 방지책: 고정 타임스텝 렌더는 prev/현재 위치 + alpha(스텝 진행률)로 보간한다
  (prevX/prevY 를 스텝 시작에 기록, game 이 alpha 노출). 약 1스텝 지연은 관전엔 무해.

### ✅ 생물 떨림(jitter) — 해결됨 (2026-06-29, 사용자 폰 확인)
- **증상**: 스프라이트가 좌우로 빠르게 진동 회전(제자리·이동 중 둘 다). 폰 실기에서만(개발 머신 비재현).
- **최종 원인(확정)**: 회전 목표를 매 스텝의 미세 이동 방향 `atan2(e.x-e.prevX, e.y-e.prevY)` 으로 잡음.
  저속/조향 흔들림이면 한 스텝 방향이 좌우로 뒤집히고, 데드존(18°)은 작은 변화만 막지 방향 "뒤집힘"
  (큰 각)은 통과 → 부들부들 회전. **위치가 아니라 회전이 주범**(제자리에서도 미세 변위 방향이 노이즈).
- **해결(커밋 62e9ac1, `render/worldView.ts`)**: 회전 목표를 **헤딩 벡터 저역통과**로. 개체별 헤딩 벡터를
  `hd += (스텝변위 - hd) * headK` 로 평활해 `sp.rotation` 이 `atan2(hd)` 를 데드존+이징으로 따라간다.
  방향이 좌우로 튀어도 평균이 0 근처로 수렴 → 각도 안정. 진짜 전환만 서서히 따라감. `?head=<0~1>` 튜닝.
- **진단 교훈(가장 중요)**: 떨림은 `?norot`(회전 고정)로 **회전/위치부터 가르는 게 첫 단추**. 사용자가
  "좌우로 진동하는 회전"이라 정확히 묘사해준 게 결정타였다. 한때 ?norot 에도 떤다고 위치로 오진해 헛돈 게
  2~3세션. 증상 묘사를 끝까지 정확히 받는 것이 진단의 핵심.
- **막다른 길(반복 금지)**: ① sim 파라미터 튠(관성 조향/목표 고정/arrive 감속/cohesion comfort/wander
  보존) ② 위치 보간(prevX/prevY+alpha) ③ 위치 저역통과(dispPos, `?smooth=0.01` 거의 얼림에도 떨림 →
  위치 진폭 무관 증명) ④ roundPixels(위치 sub-pixel용) ⑤ 회전 이징 프레임률 독립화·데드존만.
  **전부 회전 목표의 "방향 노이즈"를 못 없애 무효.** 핵심: 이징/데드존을 아무리 고쳐도 목표 자체가
  노이즈면 못 잡는다 — 목표(헤딩)를 평활해야 풀린다.

(아래는 그 진단 여정의 상세 기록 — 같은 시도 반복 방지용 보존. 라벨의 "폰 검증 대기"는 당시 기준.)
- 이미 해본 것(효과 부족 → 같은 시도 반복 금지):
  1. sim 이동을 관성 조향+목표 고정으로 부드럽게(9a70023) — sim 단계 진동은 줄었으나 화면 떨림 잔존.
  2. 렌더 위치 보간: entity.prevX/prevY + game.interpAlpha, worldView 가 prev→현재 보간(d073d1e).
  3. 회전 60fps 이징: 개체별 각도 캐시, 최단경로, 이동 임계값 0.06(7eb19df).
  → 그래도 사용자 기기에서 떨림 지속.

### 반응형 벽 회피(avoidWalls)는 진동을 만든다 — headless 프로브가 진동을 못 잡아 폰과 어긋남
- 증상: 이동 차단(지형) 후 생물이 벽에 막히면 못 돌아감. `avoidWalls`(가려는 방향이 막히면 desired 를
  통행 방향으로 회전)를 넣었더니 폰에서 끼임이 **오히려 더 늘었다**(사용자). 그런데 headless
  프로브("변위<0.1=정지")로는 끼임이 줄어든 것으로 측정돼 정반대 결론이 나왔다.
- 원인: ① avoidWalls 가 매 틱 좌우(±0.35rad)를 독립 탐색 → 벽 옆에서 선택 방향이 휙휙 뒤집혀
  desired 가 좌우로 진동 → 제자리 맴돌기. ② 반응형 조향은 오목 지형·막다른 만에서 local minima 에
  빠져 못 빠져나옴(근본 한계). ③ 프로브가 "정지"만 재서 **진동(변위는 있음)을 "안 끼임"으로 분류** —
  떨림 사건과 똑같은 맹점.
- 해결(7d653d7): 타일 격자 BFS 길찾기로 교체(`terrain.lineOfSight` 직진 우선 + 막히면 `findPath`
  경로 추종 + funnel 단축, 경로는 entity 캐시). desired 방향을 흔드는 게 아니라 "어디로 갈지"를
  격자에서 풀어 진동·local minima 가 없다. cohesion 도 무게중심이 벽 너머면 끔.
- 재발 방지책: **시각 현상(떨림/진동/맴돌기)을 "정지" 같은 단일 지표로 판단하지 말 것.** 프로브에
  방향 반전율(벽 옆 변위가 한 틱 전과 거꾸로) 같은 진동 지표를 함께 재고, 최종 판단은 폰 눈.
  정적 장애물(지형) 회피에 반응형 조향은 부적합 — 격자 길찾기가 정답.

### 우호 친척 종을 내 종 "근처"에 스폰하면 소수 밸런스가 깨진다 (근처 동거 불가)

  비용이 크다(현재 안 함).

  분리 + 상태 첫 줄에서 "단계 N/M ·" 프리픽스 제거(타임라인 막대가 대신 보여줘 중복).
- 재발 방지책: 우측 고정 위젯은 **그 줄의 좌측 텍스트가 최대로 길어져도 안 닿는지**로 자리를
  정한다. 길이가 가변인 텍스트(단계·종·시련 이름 등) 줄에는 고정 위젯을 얹지 말고, 길이가
  거의 일정한 줄(개체 수)이나 별도 영역에 둔다. 좌표로 "좌측 끝 x < 위젯 시작 x"를 계산해 마진 확인.

### 시각(이펙트)은 실제 로직과 반드시 1:1 로 일치시킨다 (전역 로직을 국소 이펙트로 그리지 말 것)
- 증상: 보스 6종 중 5종(독안개·사나운 무리·약탈자·외톨이 사냥꾼·그림자 매복자)이 **전역 솎기/흡수**
  (위치 무관, 매 틱 모든 개체를 확률로 솎음)인데, 시각은 "떼(작은 점 뭉치)"·보스 점을 특정 위치에
  그려 놨다. → "저기 떼가 있는데 왜 여기 있는 내 애가 죽지?" 로직과 이펙트가 따로 놀아 사용자가 혼란·불신.
- 원인: 로직(전역)과 시각(국소 개체)을 각각 만들고 맞추지 않음. 개체가 없는 현상(전역 재난)에 개체(점)를
  그렸다. 게다가 "보스"라는 한 이름에 개체형(추격자)과 전역 재난(독안개)을 섞어 불러 더 헷갈림.
- 재발 방지책(원칙): **새 메커니즘을 만들 때 시각을 로직과 1:1 로 설계한다.**
  · 전역 효과(위치 무관) → 화면 전체 표현(전체 틴트/맥동, 사방에서 벌어지는 kill 이펙트). 개체(점) 금지.
  · 국소 효과(특정 위치) → 그 위치에만 표현.
  · 개체가 실제로 존재하는 것만 개체로 그린다(추격자=점+즉사 반경, 도망 대상).
  · 이름도 로직에 맞춘다: 실제 개체=보스, 전역 재난=시련. "무엇이 어떻게 죽이는지"가 화면과 말에서
    일치해야 "왜 졌는지"를 안다(기획서 §7 가독성). 로직을 먼저 정하고 시각·용어를 거기에 맞춘다.

### 새 통행/감각 파라미터(canFly 등)를 배선할 때 "기존 기본값에 의존하던 호출"을 바꾸면 밸런스가 이동한다
- 증상: 날개(비행) 형질로 terrain 통행 함수에 `canFly` 파라미터를 추가하며, cohesion 판정의
  `lineOfSight` 호출도 `(…, canSwim)` → `(…, canSwim, canLand, canFly)`로 "더 정확하게" 바꿨다.
  그랬더니 내 종/야생이 날개 0(비행 안 함)인데도 독안개 통과기준 테스트가 경계에서 실패(lo=base=1).
  고산 먹이(독립 rng)를 0으로 꺼도 실패해 "고산 먹이가 원인이 아님"이 드러났다.
- 원인: 그 `lineOfSight` 호출은 원래 `canLand`를 **생략(기본값 true)**했다. `canLand`를 명시하니 물
  전용 종(swimming≥90, 물고기 떼)의 `canLand`가 false가 돼 cohesion 판정이 바뀌고(육지 너머 무리를 안
  봄), 다종 생태의 궤적이 이동 → 내 종 개체수가 경계에서 넘나들었다. 날개와 무관한 "물 전용 종 동작
  변화"가 진짜 원인.
- 재발 방지책: 새 통행/감각 파라미터를 배선할 때 **"기존에 기본값에 의존하던 호출"은 그 기본값을 유지한
  채 최소로만 바꾼다.** 정확성을 위해 인자를 더 넘기고 싶어도, 그 인자가 특정 종(물 전용 등)에서 기존과
  다른 값이면 밸런스가 이동한다. 밸런스와 무관한 부수 판정(cohesion 등)은 원래 호출을 보존하는 게 안전.
  진단 팁: **새 자원(고산 먹이 등)을 의심하기 전에 그 자원을 0으로 꺼 원인을 빠르게 가른다**(끄고도
  실패하면 그 자원이 원인이 아니다).

### 친척(우호 종)이 플레이어 능력치까지 닮으면 극단 게놈 통과기준이 흔들린다
- 증상: 친척을 시작 프리셋에 맞춰 닮게 하려고 친척 게놈을 플레이어 기반으로 블렌드(follow 0.55)했더니
  metabolism 90 같은 극단 게놈으로 통과기준을 재는 밸런스 테스트(한파 고대사 통과)가 실패. follow 를
  0.4 로 낮추자 오히려 저대사 테스트까지 겹쳐 3개로 늘었다(값 하나로 모든 극단을 못 맞춤).
- 원인: 친척이 플레이어 능력치(특히 대사)를 따라가면 극단 플레이어와 같은 환경 압력을 함께 버텨
  과경쟁 → 내 종 통과 개체수가 경계에서 떨어진다. follow 를 조절해도 고대사/저대사 극단마다 경쟁
  구도가 반대로 움직여 단일 값으로 수렴 불가.
- 해결: **능력치(속도·공격·시야·무리·대사·번식)는 플레이어와 독립(기존 50±10)으로 두고**, "친척다움"은
  이동 방식(수영·날개·초음파)과 식성 방향으로만 표현. 능력치가 플레이어 무관이면 극단 게놈에서도 친척
  경쟁이 불변 → 통과기준 보존(91 테스트 통과). 이동/감각/식성만 반영해도 "갈라진 친척" 느낌은 충분.
- 재발 방지책: 우호/경쟁 종을 플레이어 형질에 연동할 땐 **환경 적합도에 직접 걸리는 능력치(대사 등)를
  연동하지 말 것.** 정체성 형질(이동·감각·식성)만 연동하면 통과기준 밸런스를 안 건드린다.

### 소수 개체 + "닿으면 즉사 판정" 사냥에선 순수 DoT(지속 피해)·전투 형질이 개체수에 잘 안 드러난다
- 증상: 독침(venom)을 "물면 독 걸려 지속 피해(DoT)로 죽는다"로 구현했으나 프로브에서 중독사가 0, venom
  유무가 자생 개체수를 거의 안 바꿨다(venom100 atk10 ≈ venom0 atk10). 순수 DoT(즉사 안 함)로 하면 venom
  종이 사냥 완료를 못 해(무한 추격) 굶어 멸종.
- 원인: ① 사냥이 "사거리 안 매 틱 즉사 확률 판정"이라 즉사(killChanceBias 0.5 지배적)가 DoT 축적보다 먼저
  죽여 독이 쓰일 틈이 없다. ② 개체수는 채집+번식이 주 결정요인이라 사냥 효율 차이가 개체수로 잘 안 번진다
  (attack 조차 atk10→60 에서 자생 5→9로 완만). ③ 소수 개체+단일 시드는 노이즈 큼.
- 해결/타협: 독을 "즉사 확률에 누적 기여(poison/100 × venomKillBonus)"로 재설계 — venom 종은 여러 번 물어
  독 쌓아 약해진 먹이를 잡는다(attack=즉발 / venom=누적 트레이드오프). DoT 는 약하게 유지(도망친 먹이
  마무리) + 독 걸린 개체 보라 tint(지속 피해 시각 피드백). 순수 중독사는 포기.
- 재발 방지책: 소수 개체 게임의 새 전투 형질은 "개체수 우위"보다 **명확한 메커니즘 차이 + 시각 피드백**을
  노려라. 순수 시간축 효과(DoT)는 즉사 판정 사냥과 충돌하고 관찰도 어렵다. 테스트도 "개체수 우위"(노이즈)
  대신 "메커니즘 작동"(독이 걸린다·자생·사냥 발생)으로 견고히. 밸런스 세부는 폰 체감으로.

### 콜백에서 쓰는 변수는 그 콜백보다 먼저 선언해야 한다 (TDZ ReferenceError 로 부팅 전체가 블랙아웃)
- 증상: 카메라 팬/줌을 추가해 배포하니 게임이 통째로 안 뜸 — 생물·버튼·카드·로비까지 전부 안 보임(블랙아웃).
- 원인: `game.onWorldChanged` 콜백 안에 `camX = c0.x` 를 넣었는데, 이 콜백은 `game.start()`에서 곧장 호출된다.
  그런데 `let camX` 선언이 그 아래에 있어 **선언 전 접근 = Temporal Dead Zone ReferenceError**. `boot()`이
  `game.start()`에서 통째로 throw → 이후 `lobby.show()`·`app.ticker.add`(렌더 루프)까지 전부 실행 안 됨 → 완전 블랙.
- 재발 방지책: ① setup 중 즉시 호출되는 콜백(onWorldChanged 등)이 참조하는 `let/const` 는 **그 콜백보다 위에**
  선언한다. ② 타입체크·유닛테스트가 못 잡는 **런타임 부팅 오류는 `npm run smoke`**(헤드리스 chromium 으로 실제
  부팅·관전·카메라 입력을 구동해 pageerror·canvas 유무 확인)로 잡는다. **main/부팅/렌더를 크게 바꾸면 smoke 를
  돌리고 배포**한다. (이 사고는 smoke 로 즉시 잡혔을 것 — 그래서 scripts/smoke.mjs 를 추가했다.)

### 모바일 세로에서 좌하단 UI 셋이 한자리에 겹친다 (조작 열·개체 카드·드래프트 카드)
- 증상: 관전 중 좌하단 조작 열(◎ 한 마리 관찰 + 줌 ±, `zoomBar`)이 개체 정보 카드(`creatureCard`)와
  드래프트 카드(하단 전체)를 가린다. 로비에서도 이 조작 열이 떠 어색했다. 화면 소개 스크린샷 자동화
  (`scripts/screenshots.mjs`)를 만들다 겹침이 드러났다(스크린샷에선 임시로 숨겨 피했었다).
- 원인: `zoomBar`(left:6 bottom:52)와 `creatureCard`(left:8 bottom:8)가 같은 좌하단에 절대 배치인데
  **표시 조건이 없어** 로비·드래프트·개체 선택 상태와 무관하게 늘 떠 있었다. 개체 카드도 phase 가
  watch/draft 양쪽에서 그려져 드래프트 팝업과 겹쳤다. 모바일 세로는 폭이 좁아(≤540) 좌우로 못 피한다.
- 해결(main.ts): ① `zoomBar` 는 **관전(watch) + 개체 미선택(selectedId===null)** 일 때만 표시 —

- 재발 방지책: 한 모서리에 절대 배치 위젯을 새로 둘 땐 **그 자리를 이미 쓰는 위젯이 있는지** 먼저 보고,
  겹치면 "언제 보이나"를 phase/선택 상태로 **배타 제어**한다. 모바일 세로는 좌우 분할이 안 되니 같은
  모서리 위젯은 상태로 시분할한다(동시에 두 개를 같은 자리에 두지 않는다).

### 모바일 상단: 좌상단 카드의 오른쪽 끝 요소가 우상단 고정 컨트롤 밑에 깔린다 (낮밤 표식 가림)
- 증상: 좁은 폰(≤~410px)에서 정보 카드 오른쪽 끝의 낮밤 표식(점)이 우상단 컨트롤(1x·⏸) 버튼 밑에 가려 안 보임.
- 원인: 카드가 `max-width:min(94vw,300px)`로 폭 300까지 늘고, `.hud-row1`이 `space-between`이라 낮밤을 카드
  오른쪽 끝으로 밀었다. 카드 오른쪽 끝(~300px)이 우상단 컨트롤(right:12, 폭 ~101 → 왼쪽 끝 ~277px)과 겹치는데
  컨트롤 z-index(12) > 카드(8)라 컨트롤이 위를 덮는다. 넓은 폰에선 안 겹쳐 재현이 들쭉날쭉.
- 해결(hudPanel.ts): ① 낮밤을 `space-between` 오른쪽 끝 → `flex-start`로 개체 수 바로 오른쪽에 붙인다(컨트롤과
  절대 안 겹침). ② 카드 폭을 `max-width:min(300px, calc(100vw - 128px))`로 컨트롤 앞에서 끊는다.
- 재발 방지책: 좌상단 고정 패널의 **오른쪽 끝**에 정보를 두지 말 것 — 우상단 고정 컨트롤(더 높은 z-index)에 깔린다.
  space-between으로 요소를 패널 끝으로 미는 패턴은 반대편 모서리 위젯과의 충돌을 부른다. 실측(playwright
  boundingBox)으로 좌표 겹침을 확인하면 폰별 들쭉날쭉을 결정적으로 잡는다.

### 접이식 패널이 flex 컬럼(align-items:flex-start)에서도 전체 폭으로 늘어난다
- 증상: 접힌 "종 안내" 범례가 제목 한 줄뿐인데도 화면 전체 폭 바(374px)로 깔려 상단이 꽉 막혀 보인다.
- 원인: `.hud-root`가 `align-items:flex-start`인데도 범례가 콘텐츠 크기로 안 줄고 컨테이너 폭 전체를 차지했다
  (형제 `.hud-timeline-wrap`은 `align-self:stretch`라 전폭이 맞지만, 범례는 그게 없는데도 전폭). align-items만으로
  자식이 shrink-to-fit 된다고 믿으면 안 된다.
- 해결(hudPanel.ts): `.hud-legend`에 `width:fit-content`(+`max-width`)를 명시 → 접힘 84px 칩, 펼침은 종 목록에 맞춰 확장.
- 재발 방지책: 콘텐츠 크기로 줄이고 싶은 오버레이 위젯은 `width:fit-content`를 **명시**한다. align-items:flex-start만
  믿지 말 것. 실제 폭은 playwright boundingBox로 확인.

### 인라인 `style.display` 를 쓰는 패널은 CSS 클래스 규칙으로 못 숨긴다
- 증상: 드래프트 전체 화면이 뜬 동안 HUD(`.hud-root`)와 컨트롤바(`.controls-bar`)가 뿌연 유리 아래 그대로
  남아 비쳐 보였다. 그 위를 덮은 오버레이 때문에 클릭도 먹통(playwright: "draft-root subtree intercepts
  pointer events")이라 자동 검증이 먼저 걸려 넘어졌다.
- 원인: `hudPanel`·`controls` 가 가시성을 `root.style.display = ...` **인라인**으로 관리한다. 새로 추가한
  `body.draft-open .hud-root { display:none }` 는 인라인 스타일보다 우선순위가 낮아 진다.
- 해결(panelStyles.ts): 해당 규칙에 `!important` 를 붙였다.
- 재발 방지책: 전체 화면 오버레이를 새로 띄울 땐 **그 아래 깔린 위젯이 인라인 display 를 쓰는지** 먼저 확인한다.
  쓴다면 CSS 로 숨길 때 `!important` 가 필요하고, 아니면 그 위젯의 `setVisible` 을 phase 로 직접 제어한다.
  반투명 오버레이는 아래 위젯을 "안 보이게" 만들지 못한다 — 비쳐 보인다.

### 고정 크기 히어로 + `grid 1fr` + `min-height:0` → 낮은 창에서 헤더·카드를 덮는다
- 증상: 360×640 같은 짧은 폰에서 드래프트 히어로(메달리온 204×164 + 배지 + 점)가 헤더 제목 위로 올라타고
  아래로는 첫 카드를 덮었다. CTA 는 멀쩡히 보여서 "CTA 가 화면 밖으로 밀리는가"만 보는 검사로는 안 잡힌다.
- 원인: `min-height:0` 인 `1fr` 행은 콘텐츠보다 작아질 수 있고, 그 안의 고정 크기 콘텐츠는 레이아웃을 밀지
  않고 **위아래로 넘쳐흐른다**(flex 가운데 정렬이라 양쪽으로 똑같이).
- 해결(draftPanel.ts `fitHero`): 히어로를 `transform: scale()` 로 남는 높이에 맞춰 줄인다(위아래 14px 여백,
  0.4~1.4 범위). 배율은 바깥 래퍼(`.draft-hero-scale`), 등장 애니메이션(pop-soft, transform 키프레임)은
  안쪽(`.draft-hero-group`) — 한 엘리먼트에 두면 키프레임이 배율을 덮어쓴다.
- 재발 방지책: 겹침 검사는 **콘텐츠 그룹 전체**(배지·점 포함)의 boundingBox 를 헤더 아래끝·카드 위끝과
  비교한다. 대표 자식(메달리온) 하나만 보면 배지가 카드를 덮는 걸 놓친다. 낮은 창(≤640) 을 반드시 넣는다.

### 희귀도 배지는 뽑기 확률과 반드시 묶어야 한다

- 해결(cards.ts): `CARD_RARITY`(id → 등급) + `RARITY_WEIGHT`(등급 → 뽑기 가중치) 를 도입하고 `drawCards` 를

  바꾸는지** 먼저 정한다. 순전히 장식이면 그렇게 이름 붙이고, 아니면 규칙에 연결한다. `cards.test.ts` 가

### localStorage 만 진실로 삼으면 저장이 막힌 환경에서 상태가 갈라진다
- 증상: 도전 과제 테스트 4건이 "달성했는데 안 열렸다"로 실패. 원인을 보니 vitest(node)에는 `localStorage` 가
  없어 읽기·쓰기가 통째로 no-op 이었다. 브라우저 사생활 모드에서도 똑같이 터진다 — 종료 화면엔 "달성!"이 뜨는데
  대백과·로비엔 안 뜨는, 화면끼리 어긋난 상태가 된다.
- 원인: `readState()/writeState()` 가 매번 localStorage 를 직접 읽고 쓰는 구조. 저장이 막히면 방금 쓴 값도 못 읽는다.
- 해결(`game/achievements.ts`): **메모리 캐시를 진실로, localStorage 를 그 사본으로.** 첫 읽기에서 하이드레이트하고
  쓰기는 캐시 먼저 갱신한 뒤 저장을 시도한다. 저장이 실패해도 이번 세션 안에서는 일관되다(다음 실행에만 잊는다).
- 재발 방지책: 영속 상태를 새로 만들 땐 저장소를 캐시의 사본으로 다뤄라. `game/meta.ts` 도 같은 구조라 같은 결함이
  남아 있다(메타 경험치가 사생활 모드에서 세션 중에도 안 쌓인다) — 고칠 때 참고.

### 스프라이트 아래 레이어(playerG)에 그리는 표식은 몸 안쪽이면 통째로 가려진다
- 증상: 「별가루」 꾸밈이 화면에서 거의 안 보였다. 코드는 정상이고 알파도 0.95 였다.

- 해결(`render/worldView.ts`): 반짝이 시작 거리를 15px(몸 바깥)로 밀었다. 「빛나는 몸」·「무지갯빛」도 초록 지형에

- 재발 방지책: 개체 표식을 새로 그릴 땐 **어느 레이어에 그리는지** 먼저 보고, 몸에 가리지 않는 반경에서 시작한다.

### 드래프트 스크린샷은 오버레이만 걷어내면 된다(월드가 이미 멈춰 있다)
- 상황: 꾸밈을 촬영하려는데 레벨업 드래프트가 계속 끼어들어 월드를 가렸다. 일시정지 버튼은 메뉴 오버레이를 띄워
  화면을 덮고, 카드를 고르면 곧 다음 드래프트가 떴다.
- 요령: 드래프트 중에는 `world.step()` 이 멈추고 캔버스는 계속 그려진다. 그래서 `.draft-root { display:none }` 을
  주입하고 캔버스에서 `game-view-frosted` 클래스만 빼면 **정지한 무리 위에서 꾸밈 애니메이션만 도는** 깨끗한
  화면이 된다. 개체가 안 움직여 프레임 타이밍을 맞출 필요도 없다.

### 카드 설명이 sim 과 어긋나 있었다 — 「날개」는 혼자서 날지 못했다
- 증상: 「날개」 카드 설명은 "산과 바다 위를 날아 넘고"인데 실제로는 `wings 0 → 42` 라 비행 문턱(65)을 못 넘어
  **아무 일도 일어나지 않았다**. 게다가 `wings` 는 sim 전체에서 `>= flyThreshold` 문턱 비교에만 쓰여
  수치의 크기가 무의미했고, 그래서 「날개」(+42)와 「튼튼한 날개」(+30)는 사실상 같은 카드였다("날개 포인트").
- 원인: 관문/강화라는 **등급 서사만 있고 sim 근거가 없었다.** 수영·초음파·독·원거리는 모두 수치가 의미를 갖는데
  (문턱 2개, 반경, 지속 피해, 사거리) 날개만 켜짐/꺼짐이었다.
- 해결: ① 「날개」를 `flyThreshold + 3` 으로 올려 한 장으로 실제로 날게 했다. ② `flyDrainMultiplier(wings)` 를 만들어
  날개가 클수록 비행 대사가 줄게 했다(문턱에선 기존 값 그대로 → 기존 밸런스 보존). ③ 「튼튼한 날개」에
  `requiresTrait` 를 붙여 **이미 나는 종에게만** 나오게 했다. 안 그러면 못 나는 종이 골라 아무 일도 안 일어난다.
- 재발 방지책: 카드 설명에 "무엇이 된다"고 쓸 땐 **그 카드 한 장으로 그 일이 실제로 일어나는지** 수치로 확인한다.
  능력형 카드를 관문/강화로 나눌 땐 sim 이 그 수치의 크기를 실제로 읽는지 먼저 본다. `cards.test.ts` 가
  "관문 한 장으로 문턱을 넘는다"를 강제한다.

### 설명 길이에 따라 카드 레이아웃이 카드마다 달라졌다
- 증상: 드래프트 카드 중 「다산」처럼 설명이 짧은 카드만 스탯 칩이 설명과 **같은 줄 오른쪽**에 붙어, 카드마다
  레이아웃이 제각각으로 보였다.
- 원인: `.draft-card-body { flex-wrap: wrap; justify-content: space-between }`. 설명 + 칩이 한 줄에 들어가면
  나란히 놓이고, 안 들어가면 칩이 다음 줄로 넘어간다. 스펙 목업은 설명이 짧아 나란한 모습만 보여줬다.
- 해결(panelStyles.ts): `.draft-card-desc { flex: 1 1 100% }` — 설명이 항상 제 줄을 통째로 쓴다. 데스크톱은 세로
  배치라 `flex-basis` 가 높이가 되므로 `flex: 1 1 auto` 로 되돌린다.
- 재발 방지책: `flex-wrap` + `space-between` 은 **콘텐츠 길이에 따라 레이아웃이 바뀐다.** 목업의 문구 길이만 보고
  판단하지 말고, 실제 데이터(가장 짧은 것·가장 긴 것)로 확인한다. playwright 로 칩의 top/left 를 재면 결정적으로 잡힌다.

### 전체 화면 오버레이는 ?dev 패널도 덮는다(반대로, dev 패널이 오버레이를 덮는다)
- 증상: `?dev` 로 켠 디버그 패널(z-index 31)이 드래프트 화면(z-index 15) 위에 떠서 첫 카드의 이름을 가렸다.

- 재발 방지책: 전체 화면 오버레이를 만들면 **그보다 z-index 가 높은 고정 위젯이 있는지** 훑는다(hud·controls·dev).

### ▲=이득 / ▼=손해 규칙이 안 맞는 형질이 있다 (대사·식성)
- 증상: 「올빼미 눈」의 `▼ 대사 -8` 이 빨강(손해)으로 떴지만 실제로는 이득이다("기운을 아낀다").
  반대로 「뜨거운 피」의 `▲ 대사 +14` 는 초록인데 더위·가뭄에서는 손해다. 식성도 같다 — 초식/육식 중
  어느 쪽도 더 낫지 않은데 `식성 +22` 가 초록, `-22` 가 빨강으로 떴다.
- 원인: 칩 색을 **부호**로 정했다(`up ? 초록 : 빨강`). 대부분의 형질은 "많을수록 좋다"라 맞지만,
  대사는 추위↔더위 절충이고 식성은 스펙트럼이다. 어느 쪽이 이득인지는 이번 판 환경과 다가오는 대멸종이 정한다.
- 해결(`ui/traitDisplay.ts`): `NEUTRAL_TRAITS = {metabolism, diet}` + `ChipTone("gain"|"loss"|"neutral")`.
  중립은 회색(#C6B7A2)으로 칠하고 방향(▲▼)만 사실대로 남긴다. 드래프트 카드·대백과 카드 도감·내 종 팝업의
  스탯 델타 오버레이가 전부 같은 규칙을 쓴다.
- 재발 방지책: 색으로 가치 판단을 표시할 땐 **그 형질에 정말 좋은 방향이 있는지** 먼저 묻는다.
  새 형질을 넣을 때 절충형이면 `NEUTRAL_TRAITS` 에 추가한다(`traitDisplay.test.ts` 가 풀 전체를 훑어 강제).

### 사냥 판정에 쿨다운이 없어 접촉 즉시 즉사였다
- 증상(설계 결함): `stepEntity` 가 사거리에 닿는 **매 틱** 즉사 판정을 굴렸다. 호각(공격력 같음)이면 성공 확률
  0.5 라 평균 2틱(0.07초)에 죽는다. 즉 "쫓다가 붙으면 그 순간 펑". 추격이 읽히지 않았다.
- 해결(`sim/behavior.ts` · `sim/entity.ts`): `attackCd`(물기 쿨다운) + `biteOutcome(공격력, 먹잇감 공격력)` 순수 함수.
  못 죽인 물기는 기운을 깎고(`bite` 연출), 체급 차가 `biteIgnoreDiff` 넘게 밀리면 이빨이 아예 안 박힌다.
- 재발 방지책: "매 틱 굴리는 확률"은 사실상 즉시 성공이다(1 - (1-p)^n → 빠르게 1). 확률 판정에는 반드시
  쿨다운이나 1회성 게이트를 붙인다. `world.test.ts` 가 `attackCooldownTicks > 1` 과 접촉→사망 시간을 강제한다.

### 대멸종 필터가 포식 압력에 몰래 기대고 있었다
- 증상: 사냥을 약하게 바꾸자 "저대사 종은 한파를 못 넘는다" 테스트가 경계에서 깨졌다(통과기준 3인데 3마리 생존).
  추위가 아니라 **포식자가** 저대사 종을 마저 솎고 있었던 것이다.
- 해결: 파라미터 튜닝의 제약조건을 "필터가 또렷이 갈린다(고대사 통과·저대사 탈락)"로 두고 쓸었다.
  최종값(bias 0.34 · scale 1.5 · 물기 18 · 쿨다운 3)에서 필터는 오히려 더 또렷해졌다(고대사 11 vs 저대사 0).
- 재발 방지책: 포식·대사·환경 중 하나를 만지면 **나머지 둘의 통과기준 테스트를 지표로 함께 잰다.**
  프로브를 임시 vitest 파일로 짜서 `SIM` 을 런타임에 바꿔가며 쓸면 몇 분 만에 격자를 훑을 수 있다.

### 카오스 생태에서 "최종 개체 수"는 노이즈 지배 — 피크·생존시드수로 판단하라
- 증상: "큰 사냥·긴 포만"(carnGorge) 튜닝 프로브에서 같은 파라미터(B=1 R=60)가 4시드(env-1~4) 잡식
  최종 13.5, 8시드(env-1~8) 최종 8.9 로 요동쳤다. 최종값만 보면 "gorge 가 잡식을 5씩 깎는다"로 오판해
  값을 과하게 낮출 뻔했다. 실제론 잡식 최종은 시드마다 0~25 로 흩어지는 카오스 꼬리라, 파라미터 효과보다
  시드 노이즈가 훨씬 크다.
- 원인: 소수 개체 + 붐-버스트 생태는 3000틱 시점의 "최종 개체 수"가 카오스적(초기 미세차가 증폭). 평균을
  내도 시드 수가 적으면 안 수렴한다. 반면 **피크(성장 능력)**·**생존시드수(자생 여부)**는 안정적 지표다.
- 재발 방지책: 밸런스 프로브에서 종의 "건강"을 최종 개체 수 하나로 재지 말 것. ① 조기 붕괴 여부는 **피크**
  (전 구간 24~25 유지면 성장은 멀쩡, 최종 노이즈는 꼬리) ② 자생 여부는 **생존시드 n/N**(0/8→4/8 처럼
  이산적이라 노이즈에 강함) ③ 최종 평균은 방향 참고만. 하드 제약(대멸종 필터)은 어차피 단일 시드 테스트라
  그걸로 확정한다.
  맵 크기(`world.test.ts` 는 540×960, 실제 게임은 2배)를 반드시 맞출 것 — 안 맞추면 기준선도 재현이 안 된다.

### 먹잇감 조준에 "닿을 수 있는가" 검사가 없었다 (물가 머리박기)
- 증상: 균형 잡식 종이 바다 경계에서 물가에 머리를 박고 서성인다. 프로브: env-1 에서 **내 종 개체틱의 31%**가
  물속 먹잇감(대부분 물 전용 물고기)을 조준하는 데 쓰였다. 들어갈 수도 물 수도 없는데.
- 원인: `nearestFood` 는 후보 필터에서 먹이 종류·통행을 걸렀는데, 먹잇감(`chooseGoal` 의 `grid.nearestMatching`)은
  **감지되는가**만 봤다. 지형 통행 가능성을 안 봤다. 게다가 히스테리시스(`targetKeepFactor`)가 그 목표를 붙든다.
- **끼임 감지(`stuckTicks`)로는 안 풀린다.** 주석에 "물가 먹이에 억지로 들이대다 갇히는 것을 푼다"라고 적혀
  있지만, 물가에서 튕기며 진동하면 이동량이 `stuckMinMove`(0.06)를 넘어 "움직였다"로 판정된다.
  프로브로 확인: `stuckLimit` 은 2000틱 × 여러 시드에서 **한 번도 도달하지 않는다**(사실상 죽은 코드).
- 해결(`sim/behavior.ts`): `chooseGoal` 의 먹잇감 후보와 목표 유지 조건에 `terrain.isPassable(...)` 추가.
  같은 부류인 도망 판정(`computeFlee` 의 포식자 후보)에도 같은 가드를 넣었다(현재 0건이지만 육식 수생종이 생기면 터진다).
- 재발 방지책: **형제 코드끼리 필터를 맞춰라.** 먹이 후보와 먹잇감 후보는 같은 제약(감지 + 통행)을 받아야 한다.
  "안전장치가 있으니 괜찮다"고 넘기지 말고, 그 안전장치가 **실제로 발동하는지 프로브로 세라**.
  `world.test.ts` 가 "물속 먹잇감 조준 비율 < 0.5%"를 강제한다.

### 식성(diet)은 능력을 잠글 뿐, 아무 이득도 주지 않는다 (설계 공백)
- 사실: sim 전체에서 `diet` 가 하는 일은 셋뿐이다.
  `canHunt = diet > 35` · `canGraze = diet < 70` · "저 종은 포식자다"(도망 판정, `diet > 35`).
  `foodEnergy`(33)와 `predationEnergy`(36)는 **식성과 무관한 상수**다.
- 결과: 잡식(diet 50)은 사냥과 채집을 **둘 다 100% 효율로** 한다. 특화는 순손해다.
  · 「포식 본능」(식성 +22): 50 → 72 > 70 → **초식 능력 상실**. 얻는 건 공격력 +4.
  · 「초식 본능」(식성 -22): 50 → 28 < 35 → **사냥 능력 상실**. 얻는 건 번식력 +3.
  둘 다 잡식 종에겐 함정 카드다. 사용자가 "공격 +11 대신 이걸 고를 유인이 뭐냐"고 물은 그대로다.
- 프리셋도 마찬가지: 8개 중 **6개가 사실상 잡식**이다(omni 50 · scout 40 · sea 40 · sky 40 · ranged 60 ·
  **hunter 68 < 70 이라 풀도 뜯는다**). 진짜 초식은 herd(16)·venom(26)뿐이고, 진짜 육식은 하나도 없다.
- 고칠 방향(밸런스 결정 필요): 섭취 효율을 식성에 연동한다. 초식일수록 풀에서 얻는 에너지가 크고, 육식일수록
  사냥에서 얻는 에너지가 크며, 잡식(50)은 양쪽 다 중간. "제너럴리스트 페널티"라는 고전적 해법이고,
  이걸 넣어야 식성 카드와 특화 프리셋에 의미가 생긴다. 먹이 경제가 통째로 바뀌므로 대멸종 필터를 제약으로
  두고 프로브로 튜닝할 것.
- **→ 2026-07-11 해결(78598b5):** `grazeEfficiency`/`huntEfficiency` 순수 함수로 연동. 곡선을 특화 끝
  (순수 초식 ≤35 / 순수 육식 ≥70)에서 1.0 으로 평평하게, 잡식 구간(35~70)만 페널티(`dietSpecializationPenalty`
  0.3). 야생 초식·포식 생태 보존(효율 불변) + 대멸종 필터 3종 유지. 「포식/초식 본능」이 사냥/채집 효율 최대
  이득을 갖고, hunter 프리셋(68)은 사냥 98%·채집 72%라 "사냥꾼" 정체성이 효율로 뒷받침된다.

### 순수 육식 자생은 사냥 "수입" 상향으로 안 풀린다 — 병목은 사냥 성공 빈도 + 이주 부재
- 증상: 식성 효율 연동 후 순수 육식(diet 85, 채집 완전 차단) 내 종이 단독 멸종(프로브 pop 0, peak≈시작 18에서
  안 늚). 사냥 수입(huntEfficiency 보너스)을 2.3배까지 올려도, 보조 형질(speed·attack·vision↑)을 줘도 pop 0.
- 원인: 병목은 사냥 "수입"이 아니라 "성공 빈도"다. ① 야생 초식이 내 종(순수 육식)을 포식자로 보고 도망치는데
  내 종 speed 50 으론 못 따라잡는다. ② 채집 fallback 이 없어 첫 사냥 전에 굶어 죽는다(사망 원인 굶주림 16.8 vs
  포식 0.3). ③ 야생 포식자(같은 diet 85)조차 자생이 아니라 이주(immigrationFloor 4)로 연명한다 — 내 종은
  플레이어라 이주가 없다.
- 결론: 이 게임 구조(소수 개체·채집 풍부·먹잇감 도망·플레이어 이주 없음)에서 순수 육식 자생은 근본적으로
  어렵다. hunter 프리셋(diet 68, 채집 병행)이 pop 37 로 번성하는 게 정답 — "육식 빌드"의 현실적 형태는 채집을
  조금 남긴 diet 68~69 다. 순수 초식은 반대로 grazeEfficiency 1.0 + 다산·무리(herd 프리셋)로 이미 자생한다
  (병목이 포식이라 채집 효율이 아니라 개체수/속도로 푸는 것).
- 재발 방지책: 순수 육식 자생을 만들려면 사냥 수입 상수(`predationEnergy`·huntEfficiency)가 아니라 사냥 성공
  생태(먹잇감 도망 완화 / 먹잇감 밀도↑ / 플레이어 이주)를 손봐야 한다 — 야생 밸런스 대공사라 리스크 크다.
  수입 상향으로 순수 육식을 살리려는 시도는 **효과 없음이 프로브로 확인됐으니 반복하지 말 것**.

### smoke 스크립트가 Windows 에서 vite preview 서버를 orphan 으로 남긴다 (프로세스 실수)
- 증상: `node scripts/smoke.mjs` 가 exit 0 으로 끝났는데 포트 4199 preview 서버(node 2개: npx-cli + vite.js)가
  계속 돈다. 사용자 화면에 "백그라운드 작업이 90분 넘게 돌고 있다"로 보여 불안을 줬다.
- 원인: smoke.mjs 가 `spawn("npx", ["vite","preview",...])` 로 띄우고 끝에 `preview.kill()` 하는데, Windows

- 재발 방지책: smoke 를 돌린 뒤 포트 4199 를 물고 있는 node 가 남았는지 확인하고 정리한다

  **blanket kill(포트 무관 node 전멸) 금지** — 다른 세션 dev 서버까지 죽인다. 근본 고치려면 smoke.mjs 의 spawn 을
  프로세스 그룹으로 띄워(`detached:true`) `process.kill(-pid)` 하거나 tree-kill 로 자식까지 정리한다.
  (Windows: `Stop-Process -Id` 는 부모만 죽이고 detach 된 자식 vite 가 남는다 → `taskkill /PID <부모> /T /F` 로 트리째.)

### 수영 카드로 물 전용(90+)이 되면 육지 종이 땅에 갇혀 죽는다 (전환 시 재배치 없음)
- 증상: 육지에서 살던 내 종이 「지느러미」를 얻으면 뭍에서 냅다 굶어 죽는다(사용자 발견, 2026-07-11).
- 원인: swimming ≥ aquaticOnlyThreshold(90)면 물 전용(canLand=false)이 돼 육지 타일이 통행 불가가 된다.
  「지느러미」(+22)가 바다 프리셋(88)이나 「물갈퀴」(+16) 누적에 더해져 90 을 넘기면, 육지에 있던 무리가
  갑자기 물 전용이 돼 땅에 갇힌다 — 바다로도 못 가고 육지 먹이도 못 먹어 아사. 물 전용 전환 자체는 "바다
  개척자" 빌드로 의도됐으나, 런 중 카드 전환 시 개체를 바다로 옮기는 처리가 빠져 함정이 됐다.
- 해결(applyCard): 내 종 swimming 을 aquaticOnlyThreshold-1(수륙양용 상한)로 막는다(사용자 방향: "수륙양용이되
  바다까지 들어가게만"). 카드로는 물 전용이 안 되고 수륙양용(바다+육지)까지만. 진짜 물 전용은 야생 물고기
  떼만(카드 없어 상한 안 거침). **TRAIT_CEILING[swimming] 을 낮추는 방법은 안 된다** — clampGenome 을 통과하는
  야생 물고기(95)까지 잘라 물 전용을 없앤다. 카드 경로(applyCard)에서만 막아야 한다.
- 재발 방지책: 형질이 "지형 통행 자격"을 바꾸는 문턱(swimming·wings)을 카드로 넘길 때, 그 전환이 개체를 못
  가는 지형에 가두는지 본다. 통행 자격을 잃게 하는 방향(육지→물전용)은 재배치나 상한이 필요하다.

### `world.population` 은 야생 포함 **전체** 개체 수다 — 내 종은 `playerPopulation`
- 증상: 무리 방어 프로브에서 "pop 80"을 내 종 개체 수로 읽고 "다산 초식은 평상시엔 잘 산다"고 오진했다.
  실제로는 내 종이 2~5마리(8시드 중 3~6이 멸종)였고, 80은 야생 98마리를 포함한 값이었다. 그 위에서
  "이웃이 0.5명뿐"이라는 측정이 나와 코드 버그를 의심하며 한참 헤맸다 — 이웃이 없던 게 아니라
  **개체가 없었다**.
- 원인: `get population()` = `entities.length`(전 종). 승패 판정에 쓰는 건 `playerPopulation`(내 종만).
  이름이 짧고 자연스러워 프로브에서 무심코 집는다.
- 재발 방지책: **프로브에서 내 종을 볼 땐 반드시 `playerPopulation`**. "개체 수"를 찍을 때 그게 누구
  개체 수인지 한 번 더 본다. 2026-07-13 세션의 "시야 10 은 보스가 없어도 비생존 빌드" 함정과 같은 계열이다
  — 기준선·대상 없이 절대 숫자만 보면 오독한다.

### 무리 방어를 "물기 확률을 깎는" 방식으로 만들면 아무 일도 안 일어난다 (잡히는 개체는 이미 낙오자)
- 증상: "같은 종 이웃이 많을수록 사냥당할 확률이 떨어진다"를 `biteOutcome` 의 즉사 확률·피해에 저항을
  곱하는 방식으로 구현했더니, 저항이 50% 걸리는데도 잡아먹힘이 29→22 에 그치고 도달 단계·승률이 **전혀**
  안 변했다(승리 0/8 불변).
- 원인: **잡히는 개체는 이미 무리에서 떨어져 나온 낙오자다.** 무리 한가운데 개체는 애초에 잘 안 잡히고,
  잡히는 건 먹이를 찾아 홀로 나선 개체 — 그 개체는 이웃이 0~1 이라 "이웃 수" 보정이 걸리지 않는다.
  전체 개체 평균 저항(51%)만 재고 "방어가 잘 걸린다"고 오판했다. **물린 뒤가 아니라 물리기 전에** 막아야 했다.
- 해결: 표적 선택(`chooseGoal`)에서 뭉친 무리 개체를 **후보에서 통째로 제외**(이진 판정 `herdShielded`).
  사자가 물소 떼를 안 덮치는 것과 같다. 도달 2.0 → 4.0, 잡아먹힘 24 → 13.
- 재발 방지책: 확률 보정형 방어를 설계할 때 **"그 보정이 실제로 죽는 개체에게 걸리는지"** 를 먼저 본다.
  평균이 아니라 **사망 사건 시점의 값**을 재야 한다. 무리·군집처럼 "안전한 개체와 위험한 개체가 공간적으로
  갈리는" 메커니즘은 특히 평균이 거짓말을 한다.

### 초식 프리셋의 병목을 "먹이"로 지레짐작하면 안 된다 — 사인(死因)을 먼저 본다
- 증상: 다산 초식이 번식을 못 하는 걸 보고 "순수 초식이라 유한한 먹이를 야생 초식 98마리와 나눠 먹어
  에너지가 번식 문턱(78)에 못 닿는다"고 진단, **무리 채집**(뭉치면 채집 수입 +60~120%)을 만들었다.
  효과 0(도달 3.4 vs 3.6, 승리 0/8).
- 원인: 사인 집계를 보면 **굶주림 사망이 4~7 뿐**이고 잡아먹힘이 24~34 였다. 먹이는 병목이 아니었다.
  번식이 적었던 건 에너지가 모자라서가 아니라 **개체가 번식 전에 잡아먹혀서**였다. 인과를 거꾸로 읽었다.
- 재발 방지책: 프리셋·빌드가 약하면 **먼저 `world.deaths` 사인 집계를 찍는다**(단계별 개체 수 추적도 함께).
  "왜 약한가"를 추측으로 정하고 메커니즘을 만들면 멀쩡한 코드를 짜고도 아무 일도 안 일어난다.

### 새 형질을 추가할 땐 `TRAIT_KEYS` **맨 끝**에 + 종 생성에서 **rng 없이** 설정한다 (게놈 v7 에서 재확인)
- 증상(예방됨): 형질을 추가하면 `TRAIT_KEYS` 를 순회하며 rng 를 뽑는 곳(`makeKinSpecies`·`generateWildSpecies`·

- 재발 방지책: ① 새 키는 **맨 끝**에 붙인다 ② 위 다섯 곳의 "rng 없이 설정" 분기에 새 키를 **반드시 추가**한다
  (`if (key === "size" || key === "camouflage") { ... continue; }`) ③ 변이 목록(`MUTABLE_TRAITS`)의 **개수를
  유지**한다 — 독립 스트림(mutRng)이라도 소비 **횟수**가 바뀌면 개체 변이 결과가 통째로 이동한다.
  v7 은 herding 을 빼고 size 를 넣어 6개를 유지했다.
- 그리고 **새 형질의 기본값이 "중립"이 되게 설계한다**: 몸집은 50 에서 모든 보정이 정확히 0 이고(±(size-50)/50
  에 곱함) 은신은 0 이면 영향이 없다. 그래서 v7 을 통째로 얹고도 305 테스트가 그대로 통과했다. 기본값이
  중립이 아니면 새 축을 넣는 순간 기존 밸런스가 전부 이동한다.

### 새 형질의 "대가"를 여러 개 겹치면 함정 카드가 된다 (몸집이 처음에 그랬다)
- 증상: 몸집(size)에 대가를 셋(느림 0.25 · 대식 0.4 · 저번식 0.4) 걸었더니 **큰 몸이 순손해**였다 —
  몸집 85 초식이 8시드 중 7 멸종(내 종 0.6마리 vs 무방비 5.5). 잡아먹힘은 27→15 로 줄어 방어는 분명히
  작동했는데, 느려서 먹이를 못 찾고 · 많이 먹어 굶고 · 새끼로 메우지도 못하는 **삼중고**가 이득을 삼켰다.
- 원인: 대가 셋이 **곱해져서** 걸린다는 걸 간과했다. 하나하나는 온건해 보여도 합치면 종이 못 산다.
  특히 초식은 "번식으로 사는" 종이라 번식 페널티가 치명적이다.
- 해결: 대가를 절반 가까이 낮추고(0.08 / 0.12 / 0.18) 방어 이득(sizeBiteWeight)을 0.8 → 1.4 로 키웠다.
  무방비 6.6 → 몸집 85 는 9.0 · 몸집 100 은 22.6(잡아먹힘 24→5).
- 재발 방지책: 형질 하나에 대가를 여러 개 걸 땐 **각각을 온건하게 잡고 프로브로 합산 효과를 먼저 잰다.**
  "이득 하나 vs 대가 셋"은 거의 항상 대가 쪽이 이긴다.
