### GitHub Pages 첫 배포 시 configure-pages 권한 에러
- 증상: `actions/configure-pages@v5` 가 "Resource not accessible by integration" 로 실패,
  Pages 사이트 생성 안 됨.
- 원인: 워크플로 기본 `GITHUB_TOKEN` 권한으로는 Pages 사이트를 처음 만들지 못함.
- 재발 방지책: 레포 생성 직후 사용자 토큰으로 Pages 를 먼저 활성화한다 →
  `gh api --method POST repos/<owner>/<repo>/pages -f build_type=workflow`
  (Git Bash 는 leading slash 를 경로로 오인하니 슬래시 없이). 그 후 워크플로 재실행.

### sim 행동을 바꾸면 개체 수 스케일이 변해 절대 기준 밸런스 테스트가 깨진다
- 증상: 이동 로직(관성/목표 고정)을 매끄럽게 했더니 채집 효율이 올라 개체 수가 전반 상승,
  한파 대멸종 필터의 절대 통과기준(저대사<10) 테스트가 실패(저대사가 14로 생존).
- 원인: 게임 밸런스가 "절대 개체 수 기준"으로 잡혀 있어, 행동 효율이 바뀌면 분포가 통째로 이동.
- 재발 방지책: sim 행동을 바꾸면 무작정 상수만 만지지 말고 **임시 프로브 테스트로 분포부터 측정**
  (forage/poison/cold/heat/chaser/sanity 한 번에 console.log) 후 타깃 튠. 평상시에 영향 없는
  지점(예: 대멸종 전용 배율)을 골라 고치면 sanity/Phase3 회귀를 피한다.

### 종 다양성: 단일 먹이 자원에서 경쟁 배제로 금방 1~2종만 남음
- 증상: 야생종을 늘려도(3→6종) 수천 틱이면 1~2종(보통 내 종)만 남음. 시드 편차 큼.
- 원인: 모든 초식종이 같은 먹이 한 종류를 두고 경쟁 → 경쟁 배제 원리상 한둘만 살아남음.
  36마리로 시작하는 내 종이 동등 채집력으로 수적 지배까지 함.
- 완화책(현재): 생존 전략 분리(다산/저대사 장수/잡식/포식) + 야생 영역(클러스터) 스폰으로
  초·중반 공존을 늘림. 후반 붕괴는 못 막음.
- 근본 해결(적용 478e1d1): (1) 먹이 종류 분할 — food.kind 3종, 야생 초식종마다 다른 종류 전문이라
  서로 경쟁 안 함(전문종 공존). (2) 야생 주기적 이주 — 적은 야생종을 10초마다 소수 보충(멸종 바닥).
  결과: 4시드 전부 t750~t4000 내내 7종 전부 유지. 교훈: 단일 자원 경쟁 배제는 자원을 쪼개야 풀린다.

### 렌더 보간 없으면 sim 을 부드럽게 해도 화면은 드득거린다
- 증상: 행동 로직을 매끄럽게 고쳤는데도 생물이 잔떨림(특히 정지 화면/로비에서).
- 원인: sim 은 30스텝/초, 화면은 60fps. 보간 없이 sim 위치를 그대로 찍으면 두 프레임마다 점프.
- 재발 방지책: 고정 타임스텝 렌더는 prev/현재 위치 + alpha(스텝 진행률)로 보간한다
  (prevX/prevY 를 스텝 시작에 기록, game 이 alpha 노출). 약 1스텝 지연은 관전엔 무해.

### ❗생물 떨림(jitter) 미해결 — 사용자 기기에서 지속 (최우선 과제)
- 증상: 수많은 생물이 화면에서 드득거린다. 인게임·로비(메인) 모두. 느린 종(들풀=보라, 거북=초록)이
  특히 심하다고 했으나 최근엔 "수많은 생물"로 광범위. 폰 실기에서만 확인(개발 머신/테스트로 재현 안 됨).
- 이미 해본 것(효과 부족 → 같은 시도 반복 금지):
  1. sim 이동을 관성 조향+목표 고정으로 부드럽게(9a70023) — sim 단계 진동은 줄었으나 화면 떨림 잔존.
  2. 렌더 위치 보간: entity.prevX/prevY + game.interpAlpha, worldView 가 prev→현재 보간(d073d1e).
  3. 회전 60fps 이징: 개체별 각도 캐시, 최단경로, 이동 임계값 0.06(7eb19df).
  → 그래도 사용자 기기에서 떨림 지속.
- 최근 시도(3b6c961, **폰 검증 대기** — 효과 확인 전이라 아직 "실패" 목록 아님):
  새 가설 = **회전 이징이 프레임률 의존**이었다. `ang += diff*0.2`가 프레임당 고정이라
  120Hz 폰에선 한 sim 스텝(33ms)에 4프레임이 들어가 60Hz(2프레임)보다 회전이 ~2배 빠르게
  방향 노이즈를 쫓음 → 느린 종이 더 떨림. 개발 머신(60Hz) 비재현과 일치.
  수정: (a) 이징을 dt 기반으로 프레임률 독립화(`1 - (1-0.2)^(dt/16.67)`),
  (b) 회전 갱신을 실제 이동 0.35px/스텝 초과일 때만(느린 종 노이즈 억제).
  → 사용자: 그래도 떨림 지속. "좁은 각도 안에서도 떨린다, 전 종(내종/노랑/파랑) 다."
- 다음 시도(5bfe532, **폰 검증 대기**): 핵심 깨달음 = **이징을 아무리 고쳐도 매 스텝 회전
  목표가 미세하게 흔들리면 그걸 계속 쫓아 떤다.** → **각도 데드존(방향 굳히기)**: 좁은 각도
  (기본 ~18°) 안의 방향 변화는 무시하고, 데드존을 넘는 진짜 전환만 이징. 사용자가 제안한
  "한 번 방향 굳히면 잠깐 못 돌게"를 각도 기반으로 구현.
- 폰 실기 진단·튜닝 토글(URL 쿼리, src/debug.ts):
  · `?norot`(회전고정) — 이래도 떨리면 회전 아님(위치보간/서브픽셀 의심, 가설 B/E).
  · `?nointerp`(보간끔), `?showalpha`(alpha 0~1 변하면 보간 작동 → 가설 B 배제).
  · `?dz=<라디안>`(데드존 크게=덜 떨고 덜 돌아봄), `?rotk=<0~1>`(이징 세기). 좌하단 배지에 현재값.
  데드존을 키워도 안 잡히면 회전이 주범이 아닐 가능성 → `?norot` 로 확정할 것.
- **결정적 진단(사용자): `?dz=0.7`·`?norot`(회전 완전 끔)에도 "제자리에서 부들거림" → 회전이
  아니라 위치(position) 진동이었다.** 평상시 카메라는 정적(zoom=1)이라, 화면 픽셀이 떨리려면
  sim 안에서 위치가 실제로 미세 진동하는 것뿐. (교훈: 회전만 의심하다 두 세션 허비 — 떨림은
  `?norot` 로 회전/위치부터 가르는 게 첫 단추.)
- 위치 진동의 원인·수정(bf242cb, **폰 검증 대기**): `wanderDesired` 가 매 틱 헤딩을
  `atan2(현재속도)±0.5rad` 로 재추첨. 먹이 종류 분할로 자기 먹이가 시야 밖이라 배회 개체가 많고,
  느린 종은 속도 방향이 노이즈 → 매 틱 방향 급변 = 제자리 random walk. 보간은 스텝 경계 급변을
  못 메운다. → 헤딩을 개체에 보존(`entity.wanderAngle`)하고 매 틱 ±0.15rad 만 누적(부드러운 떠돌기).
  · UI: 빌드패널↔컨트롤바 우상단 겹침 → 빌드패널 top:62 로 내림. 디버그 배지는 HTML 좌하단으로
    옮겨야 패널에 안 가린다(전엔 Pixi 텍스트라 가려짐).
- 다음 시도(bf25e9f, **폰 검증 대기**): 사용자 "?nointerp 에도 부들거림 = 먹이/먹잇감 오버슈트".
  원인 = `desired` 가 항상 maxSpeed(감속 없음) → 목표를 지나쳐 진동. → **도착 감속(arrive)**:
  목표가 arriveRadius(18px) 안이면 거리 비례로 감속해 안착(무리 cohesion 도 동일). 사냥은
  전속 유지(포식 밸런스). `toward()` 헬퍼. 핵심 교훈: 조향에 가속만 있고 **감속(arrive)이 없으면**
  목표 주변에서 영원히 오버슈트한다.
- 다음 시도(804cd62, **폰 검증 대기**): 사용자 "보라/노랑/파랑이 최악" → 정확히 herding 최고 3종
  (들풀0.6·풀벌레0.72·초식0.6). **무리 cohesion 이 주범 확정.** 원인: `neighborhood` 의 무게중심(COM)을
  3×3 격자 블록(240px)으로 계산 → 생물이 80px 셀 경계를 넘을 때마다 블록이 한 칸 이동해 포함 개체가
  바뀌며 COM 이 매 틱 튐(자기 칸 기준 + 전 종 포함이라 더 흔들림). 무리 종이 이 튀는 COM 을 매 틱
  쫓아 제자리 떨림. → **cohesion comfort zone**: COM 에서 herdComfortRadius(20) 안이면 cohesion 0,
  벗어난 정도로 ramp(26) 동안 0→최대(낙오자만 모음). huddle(개체수 기반)은 불변→추위 밸런스 유지.
  교훈: 격자 기반 이웃 질의의 COM 은 셀 경계에서 양자화 점프한다 — 그걸 직접 추종하면 떨린다.
- 유력 가설(다음 진단 순서):
  A. 회전 떨림 주범 가능성: 회전 목표(스텝 이동방향)가 느린·군집(herding) 개체에선 본질적으로 노이즈가
     큼 → 이징(0.2)이 노이즈를 계속 쫓아 미세 회전 지속 → 몸이 길쭉해 떨림으로 보임. **회전을 잠시
     고정해 떨림이 사라지는지부터 확인**(회전/위치 切分). 해결안: 속도 충분할 때만 회전 갱신(임계값↑),
     강한 댐핑, 또는 여러 프레임 누적 변위 방향으로 회전.
  B. 위치 보간 무효 가능성: interpAlpha 가 늘 0/1이면 보간이 안 먹음. 화면에 alpha 디버그 출력해
     0~1로 변하는지 확인. 로비(ambientAcc)/관전(acc) 경로가 달라 둘 다 점검.
  C. 고주사율(120Hz) 기기에서 ticker·deltaMS 상호작용(자기보정되지만 실기 확인).
  D. 스프라이트 풀이 index 기반(id 아님): 죽음/번식/이주로 entities 순서가 매 스텝 바뀜. 최종 이미지는
     맞지만 텍스처/회전 캐시 엇갈림 점검(회전 캐시는 id 키라 OK로 추정).
  E. 월드 컨테이너 비정수 scale-to-fit → 서브픽셀 시머. roundPixels 등 점검.
- 디버그 계획(실기 전용): 임시 토글 3개 — (1) 회전 고정, (2) 위치보간 on/off, (3) alpha 표시 — 를 넣어
  폰에서 어느 것이 원인인지 切分한 뒤 타깃 수정. 재현이 실기에서만 되므로 토글 기반 진단이 핵심.
