# 적자생존 (Selection Pressure)

한 종(種)을 길러 생태계 정점에 올리는 **관전형 진화 로그라이크**.
플레이어는 직접 조작하지 않고 라운드 사이에 카드를 **선택만** 하며, 규칙기반 생태 시뮬이
그 결과를 살아 움직이게 한다. 기획 전문은 [`적자생존_기획서_v0.1.md`](적자생존_기획서_v0.1.md).

- **스택:** TypeScript + PixiJS v8 + Vite. 모바일·세로 우선, 정적 URL 배포.
- **현재 단계:** Phase 0 완료(스캐폴드) → Phase 1(시뮬 코어) 착수 전.
- **문서 지도(무엇이 어디서 단일 진실인지):** [`docs/INDEX.md`](docs/INDEX.md)

## 코딩 컨벤션 (TypeScript + PixiJS)

> 이전 프로젝트의 Godot/GDScript 규칙은 이 프로젝트에 해당 없음. 아래가 이 프로젝트의 규칙이다.

### 가장 중요한 구조 규칙 — 시뮬과 렌더 분리
- **`src/sim/` 는 순수 TypeScript다. PixiJS 를 절대 import 하지 않는다.**
  - 시뮬은 화면 없이 돌아가고, 단위 테스트가 가능하며, 결정론적이어야 한다.
  - Pixi 는 "그리기" 만 담당한다 → `src/render/` 에서만 sim 의 상태를 읽어 그린다.
- **결정론: `Math.random()` 사용 금지.** 시뮬 안의 모든 무작위는 시드 RNG(`src/sim/rng.ts`)를
  주입해 쓴다. `(게놈 + 환경 시드) → 항상 같은 결과` 가 깨지면 재현성·디버깅·비동기 생물 공정성이
  전부 무너진다. (기획서 §3.4)
- **게놈은 버전 붙은 직렬화 구조를 유지한다.** 형질을 추가/변경하면 `GENOME_VERSION` 을 올리고
  `migrateGenome` 에 마이그레이션 단계를 더한다. (기획서 §3.1)

### 타입
- **`strict` 전체 + 추가 엄격 옵션 on** (`noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`,
  `noUnusedLocals/Parameters`, `verbatimModuleSyntax` 등 — `tsconfig.json` 참조).
- **모든 함수 반환값·공개 변수에 타입 명시.** `any` 금지(불가피하면 `unknown` 후 좁히기).
- `verbatimModuleSyntax` 때문에 **타입만 가져올 땐 `import type`** 을 쓴다.
- 배열/객체 인덱싱 결과는 `noUncheckedIndexedAccess` 로 `T | undefined` 다 — 꺼낼 때 좁히거나
  단언(`as T`)을 의식적으로 한다.

### 모듈/경로
- import 경로 별칭 **`@/` = `src/`** (`vite.config.ts` + `tsconfig.json` 양쪽에 설정됨).
- 새 시스템은 역할별 디렉터리에: `sim/`(시뮬 로직) · `render/`(Pixi 렌더) · `game/`(런 구조·드래프트) ·
  `ui/`(UI). 자세한 구조는 [`docs/INDEX.md`](docs/INDEX.md).

### 검증 (작업 끝낼 때)
- `npm run typecheck` · `npm test` 통과 확인. sim 로직을 건드렸으면 해당 테스트도 추가/갱신한다.

## UI 문구 규칙 (한국어, 사용자에게 보이는 모든 멘트)

해설·안내·예고 등 사용자에게 보이는 한국어 문구는 다음을 지킨다.

- **한글 사이 em dash(—) 금지.** 마침표·쉼표·줄바꿈으로 대신한다.
- **쉬운 말만 쓴다.** 생태/진화 전문 용어는 뜻을 풀어 쓴다:
  - 적합도 필터(fitness filter) → "잘 적응한 종만 통과하는 관문"
  - 개체군(population) → "개체 수" 또는 "무리"
  - 대멸종(mass extinction) → 그대로 쓰되 "환경이 통째로 바뀌는 큰 시험"으로 풀이 가능
  - 포식 → "잡아먹기"

  게임 기본 용어 **"형질" · "게놈" · "드래프트"** 는 그대로 쓰되, 처음 나올 때 한 번 풀이한다.
- **"이겨요/이겼어요" 대신 "승리"** 를 쓴다.
- **보스 위협 예고는 반드시 읽혀야 한다.** "왜 졌는지 모르는데 졌다"가 되지 않도록, 게놈에서
  자동 생성하는 위협 요약도 위 규칙을 따른다. (기획서 §4.2)

### 톤·문체 (미확정 — 임의로 정하지 말 것)
예전 "차분한 자연 관찰 다큐 / 야외 도감(Attenborough류)" 톤 가이드는 **제거했다.** 사용자가 정한 게 아니라
내가 레퍼런스를 찾아 임의로 정한 페르소나였고, 이 게임(긴장·위협·재시작이 있는 로그라이크) 정체성과
맞지 않는다. (2026-07-09 사용자 지시)
- **문구 목소리(톤)는 사용자와 합의해 정한다.** 임의로 다큐 내레이션·관찰자 페르소나 같은 걸 덧씌우지 않는다.
- 톤이 정해지기 전까지는 위의 **쉬운 말·용어·읽히는 예고** 규칙만 지키고, AI가 쓴 듯한 나열식·과설명·근거
  없는 일반화만 피한다.
- (게임 모델 자체는 별개다 — 플레이어가 직접 안 굴리고 **선택만** 하는 관전형이라는 건 기획서의 설계이지
  "톤"이 아니다. 그건 유지.)

## 운영 규칙 (세션 루틴·커밋·문서)

### 세션 시작 루틴 (본격 작업 전 가장 먼저)
다음을 읽고 "직전까지 한 것 / 다음 후보 / 추천 한 가지"를 정리해 제안한 뒤 시작한다:
0. **`ACTIVE_WORK.md`(작업 조율판) — 가장 먼저.** 다른 세션이 claim 한 범위를 피해 내 범위를 정하고
   내 블록을 추가한 뒤 시작한다(아래 "동시 세션 루틴" 참고). 파일이 없으면 동시 세션이 없는 것.
1. [`docs/design/backlog.md`](docs/design/backlog.md) — 다음 작업의 **단기** 후보(당장 할 것)
2. [`docs/ROADMAP.md`](docs/ROADMAP.md) — **장기 형질/시스템 로드맵**(P3 감각·P4 이동 형질·P5 전투 등).
   backlog 에 없는 큰 그림이 여기 있다. **추천할 때 backlog 만 보지 말고 ROADMAP 의 다음 단계도
   반드시 후보로 올린다** — 여기 적힌 초음파·날개/지느러미/다리·독침/원거리 같은 형질 확장을 놓치지 말 것.
3. 최신 [`session_logs/YYYY-MM-DD.md`](session_logs/) — 직전 세션 상태·결정·미해결
4. [`docs/design/known_issues.md`](docs/design/known_issues.md) — 반복 금지 함정/오류
5. `git log --oneline -10` + `git status` — 누적 변경
사용자가 곧장 작업을 지시하면 그것부터 하되, 위 파일로 맥락을 먼저 맞춘다.

### Git commit / push 루틴
**커밋은 작은 단위로 자주(로컬), push 는 중요한 마일스톤마다 몰아서 한다.** (2026-07-05 사용자 지시)
- 이유: push 가 GitHub Pages 배포를 트리거하는데, 짧은 간격으로 여러 번 push 하면 배포 큐가 엉켜
  실패("Deployment failed")가 쌓이고 **사용자에게 fail 메일이 계속 온다**(known_issues 참조). 커밋
  자체는 배포를 안 건드리므로 로컬 커밋은 자유롭게 자주 하되, push 만 묶는다.
- 즉: 의미 있는 작업 단위가 끝날 때마다 **로컬 commit**(자동). **push 는** 기능 하나가 온전히
  끝나는 등 "중요한 부분"에서 그동안 쌓인 커밋을 **한 번에 push origin main**. 자잘한 커밋마다
  push 하지 않는다.
- push 한 뒤엔 `gh run watch` 로 배포 성공을 확인하고, stuck 이면 `gh run rerun <id> --failed`.
- 커밋 메시지: 한국어, `prefix(scope): 설명`(feat/fix/refactor/docs/chore), 이모지 없음,
  본문은 변경 항목 bullet.
- `main`에 직접 push (이 프로젝트는 branch/PR 안 씀).
- destructive 작업(force push, reset --hard 등)은 먼저 확인.

### 동시 세션 루틴 (여러 세션 병행 시)
사용자는 여러 터미널(PowerShell)에서 **여러 세션을 동시에** 돌릴 수 있다. 각 세션은 자기 작업
파일에만 손대고, 다른 세션 작업을 절대 휩쓸지 않는다.

**작업 조율판 `ACTIVE_WORK.md` (겹침 방지의 단일 진실, git 미추적·로컬 실시간 공유):**
- **작업 시작 전 반드시 읽는다.** 다른 세션이 claim 한 범위(파일/디렉터리)는 **편집 금지**(읽기 OK).
  겹치지 않는 곳으로 내 범위를 정한다.
- 내 작업을 **내 블록으로 추가**한다 — 시작 시각·예상 종료(ETA)·범위·작업 내용 명시.
  **자기 블록만 편집**(고유 id 의 `<!-- BEGIN/END session: <id> -->` 마커).
- 범위가 바뀌면 내 블록을 갱신하고, **작업이 끝나면 내 블록을 삭제**한다(커밋·푸시 직후).
- 파일이 없으면 새로 만든다. ETA 가 한참 지났는데 흔적이 없으면(중단 추정) 사용자에게 확인 후
  정리한다 — 임의로 지우지 않는다.
- 이 파일은 절대 커밋하지 않는다(`.gitignore` 에 있음). 같은 머신의 여러 터미널이 같은 워킹
  트리를 보므로 git 없이 즉시 공유된다.

- **`git add -A`/`git add .` 금지.** 다른 세션의 진행 중 변경까지 스테이징한다. 내가 바꾼 파일만
  **경로로 명시 스테이징**한다(`git add <내 파일> ...`).
- 커밋 전 `git status` 확인: **내가 안 바꾼 파일이 modified 면 다른 세션 작업**이다. 그 파일은
  스테이징·수정·`reset`·`checkout`·revert 하지 않는다(읽기는 OK).
- push 거부(non-fast-forward) 시: `git pull --rebase --autostash origin main` 후 다시 push.
- 공유 문서(`session_logs/`·`backlog.md`·`known_issues.md`)는 **끝에 append**만 한다.
- **공유 dev 서버 주의**: 포트 무관 블랭킷 kill(`pkill -f vite` 류) 금지 — 다른 세션의 dev
  서버까지 죽인다. 내 서버만 그 PID 로 정리하고, 죽었으면 다시 띄워 사용자 화면을 복구한다.

### 오류 기록 루틴 (반복 방지)
세션 중 버그·설계 함정·작업 실수를 발견하면 [`docs/design/known_issues.md`](docs/design/known_issues.md)에
"증상 → 원인 → 재발 방지책"으로 기록한다. 게임 버그뿐 아니라 프로세스 실수(도구 오용, 커밋 누락
등)도 포함. 세션 시작 루틴에서 이 파일을 먼저 읽어 예방한다.

### 세션 로그
매 세션 종료 시 `session_logs/YYYY-MM-DD.md`(날짜별 새 문서)에 요약·주요 변경·결정·미해결을
기록한다. 같은 날짜면 이어서 추가(`## 세션 2` 등으로 구분). 보통 20~60줄.

### 구현 계획 문서는 "앞으로 할 일"만 남긴다 (중요)
[`docs/design/backlog.md`](docs/design/backlog.md)·[`docs/ROADMAP.md`](docs/ROADMAP.md) 같은
**구현 계획 문서에는 앞으로 할 일·필요한 일만** 남긴다. 완료됐거나 계획이 바뀐 항목은 **지운다**
(✅ 달성 목록을 쌓지 않는다). 무엇을 왜 했는지의 기록은 `session_logs/`(+ 필요 시
`known_issues.md`)에만 남긴다. 작업을 끝낼 때마다 해당 항목을 계획 문서에서 삭제한다.
