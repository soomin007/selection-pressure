## 0. 한 줄 피치

> 한 종(種)을 길러 생태계의 정점에 올리는 진화 로그라이크. 직접 조작하지 않고 **선택만** 하면, 규칙기반 생태 시뮬이 그 결과를 살아 움직이게 한다.

**코어 판타지:** *나는 컨트롤러가 아니라 사육사/진화의 손이다. 내 선택이 한 혈통의 운명을 결정하고, 나는 그 결과를 지켜본다.*

## 1. 게임 개요

### 1.1 장르
적자생존 × 오토배틀러 × 로그라이크 + 생태계 시뮬 (관전형)

### 1.2 핵심 설계 원칙
이 게임은 여러 게임을 옆으로 붙인 게 아니라, **하나의 판타지를 여러 겹으로 쌓은 것**이다. 각 레퍼런스는 별도 모드가 아니라 레이어다.

| 레이어 | 레퍼런스 | 역할 |
|---|---|---|
| 콘텐츠 (뭘 하는가) | 스포어 / Everything is Crab | 한 종을 진화시킨다 |
| 조작 모델 | 오토배틀러 / TFT | 직접 안 굴리고 **선택만** = 사육사지 컨트롤러가 아님 |
| 세션의 형태 | 로그라이크 | 한 혈통이 살아남거나 멸종한다 = 긴장 + 재시작 |
| 확장 레이어 | Brotato식 언락 | 시간이 갈수록 드래프트 풀이 넓어진다 (스트레치) |
| 구조 모델 | Slay the Spire | 막마다 보스 게이트, 사이를 카드로 빌드업 |

### 1.3 핵심 차별점
- **Everything is Crab / Brotato는 직접 조작 액션 게임.** 이 게임은 **관전형 시뮬**이다. 전투는 "벌어지는 순간"이 아니라 **드래프트한 시점에 이미 승패가 갈린다.**
- 따라서 *반사신경 게임*이 아니라 *빌드/적응 게임*이다.

### 1.4 타깃 플랫폼
모바일 웹 우선 (정적 URL 배포). 데스크톱 브라우저는 레터박스로 대응. (§9 참조)

## 2. 코어 게임플레이 루프

### 2.1 조작 모델
- 플레이어는 개체를 **직접 조작하지 않는다.**
- 라운드 사이에 **선택(카드 드래프트)** 만 하고, 시뮬레이션이 자동으로 돌아간다.
- 플레이어는 그 결과를 **관전**하며, 다음 선택에 반영한다.

### 2.2 라운드 구조 (런 = 한 혈통의 일생)
```
[드래프트] → [시뮬레이션 관전] → [결과 정산] → [드래프트] → ... → [보스 게이트]
```

### 2.3 카드 드래프트
- 매 라운드 **3장 중 1장 선택.**
- 뽑히는 후보는 **랜덤** → 운(luck) 요소 포함 (원안 아이디어 유지).
- 카드는 **런 내에서는 영구** (게놈/행동에 누적 적용), **런이 끝나면 리셋** (로그라이크).

## 3. 시스템 상세

### 3.1 게놈 (Genome) — ⭐ 가장 중요한 데이터 구조
- 한 종을 정의하는 **형질값 묶음**: 속도 / 공격 / 시야 범위 / 무리 성향 / 대사율 / 번식률 / 식성 등.
- **⚠️ 처음부터 직렬화 가능(serializable)하고 버전이 붙은(versioned) 데이터 구조로 설계할 것.**
  - 이유: 나중에 비동기 생물 기능(§6)을 붙일 때 게놈을 그대로 들어 올려 네트워크에 실으려면 이 forward-compatibility가 필수. Phase 1에서 거의 공짜로 챙길 수 있으니 지금 챙긴다.
  - 예: `{ "genomeVersion": 1, "traits": { "speed": 0.7, ... } }`

### 3.2 환경 (Environment)
- **절차 생성** 맵 (먹이 분포 · 위험 · 기후).
- 매 판마다 다르게 생성 = 로그라이크 변주.
- **⚠️ "환경이 매 판 다르다"는 기계적으로 의미가 있어야 한다:** 환경마다 유리한 게놈이 달라져야 드래프트가 진짜 결정이 된다. (예: 추운 환경 → 대사·보온 형질 유리, 개활지 → 속도·시야 유리)

### 3.3 시뮬레이션 엔진 — 규칙기반 (NOT 머신러닝)
- 개체 지능은 **ML이 아니라 규칙기반(utility AI + boids/steering)** 으로 구현한다.
- "지능처럼 보이는 것"은 단순 규칙 × 환경의 상호작용에서 창발한다 (보이드 / 라이프게임 / 스포어 라이트).
- 장점: 가볍고, 클라이언트에서 전부 돌고(서버·API·비용 0), 디버깅 가능.
- 개체는 매 틱마다 게놈 + 단순 행동 규칙으로 행동: 무리짓기(steering), 채집, 도망, 전투.

### 3.4 결정론 / 시드 고정
- 모든 RNG는 **시드로 고정**한다.
- `(게놈 + 환경 시드) → 항상 같은 결과.`
- 효과: 로그라이크 재현성, 디버깅, 그리고 나중에 비동기 생물 비교의 공정성이 공짜로 따라온다.

## 4. 승리 조건 & 보스

> **결정:** "과반 지배" 원안은 폐기. 사실적 생태 시뮬에서 한 종의 과반 점유는 먹이 붕괴 → 개체수 폭락으로 자멸하기 쉬워, 밸런싱이 칼날 위를 걷게 된다. 대신 **Slay the Spire식 보스 게이트** 채택 — 정밀한 창발 밸런싱 없이 "정해진 관문 통과" 여부로 판정하므로 솔로 개발에 훨씬 tractable.

### 4.1 보스 게이트 (슬더스 스타일)
- 여러 종류의 보스를 막마다 배치.
- **⚠️ 한 보스에 모든 강점을 몰아넣지 말 것.** 그러면 단일 정답 카운터(드래프트 무의미)거나 깰 수 없는 벽이 된다.
- 대신 **위협 축이 다른 보스**를 둔다:

| 위협 축 | 예시 |
|---|---|
| 속도형 | 빠른 추격자 |
| 장갑형 | 단단한 방어 |
| 무리전술형 | 떼로 몰려옴 |
| 독/상태이상형 | 지속 피해 |

→ 플레이어의 "특화할까 / 헷지할까" 드래프트 결정이 살아난다.

### 4.2 사전 예고 (필수)
- 관전형 게임이므로 보스는 **전투 전에 위협 프로필을 반드시 예고**한다.
  - 안 그러면 "왜 졌는지 모르는데 졌다" = 카운터 불가능한 RNG 벽처럼 느껴진다.
- **⚠️ 구현 일감:** 임의의 게놈 → 위협 요약 텍스트 자동 생성기 (예: "빠른 무리사냥형 + 독"). 게놈→설명 변환은 별도로 만들어야 함.

### 4.3 최종 보스 = 대멸종 (Mass Extinction)
- 마지막 게이트는 개별 몬스터가 아니라 **격변/대멸종 이벤트**로 빌드업한다.
- 이것은 **환경 적합도 필터(fitness filter)** — 잘 적응한 개체군만 통과한다.
- 테마적 근거: 진화생물학에서 궁극의 적자생존 시험이 곧 대멸종이다 (공룡 멸종 등). 테마 · 메커닉 · 클라이맥스가 한 점에서 만난다.

### 4.4 패배 조건
- 내 종의 멸종 (개체수 0).

## 5. 메타 진행 (스트레치, Phase 1 아님)

- Brotato식 **수평 언락**: 런을 완료할 때마다 새 형질/카드를 잠금 해제 → **드래프트 풀 확장.**
- **⚠️ 수직(파워) 언락 금지.** 로그라이크 위에 수직 메타를 얹으면 초반 런이 비참해지고 후반엔 시시해진다. 언락은 power가 아니라 **option(수평)** 이어야 한다.
- **⚠️ Phase 1 필수 아님.** 언락 하나하나가 재밌고 밸런스 맞는 새 형질을 요구해 콘텐츠를 많이 잡아먹는다. "언락 없이 한 판이 재밌다"가 플레이테스트로 확인된 다음 retention 레이어로 얹는다.

## 6. 비동기 생물 / 플레이어 생물 보스 (스트레치)

> 클리어한 생물 자체를 다른 런의 등장 풀에 넣는 기능. 아키텍처가 게놈+시드라 ML도 실시간 네트코드도 필요 없다 — 게놈 직렬화 업로드/다운로드면 끝. (계보: 스포어 / 섀도우 오브 모르도르 네메시스 시스템 / 포르자 Drivatar)

### 6.1 현재 방침 (서버 전): 수동 큐레이션
- 완전 자동화는 서버 연결이 필요 → 추후.
- **그 전까지는 직접 검토 후 등장 풀에 수동 등록.** (콜드 스타트 대응 = 개발자 제작 생물 풀 선탑재와 동일 메커니즘)

### 6.2 풀어야 할 문제 (구현 시)
- **밸런스/티어링:** 클리어한 생물 = "이길 만큼 최적화된" 빌드. 그대로 떨구면 거저 주는 승리거나 못 깨는 벽. → **어느 난이도에서 클리어했는지로 티어를 매겨 적절한 지점에 서빙.**
- **콜드 스타트:** 출시 초기엔 유저 생물이 없음 → 개발자 생물 풀 선탑재 필수. 이 기능이 코어 경험의 필수 부품이 되면 안 됨.
- **모더레이션:** 업로드 데이터가 주로 숫자(게놈)라 어뷰징 표면 작음. 단 **커스텀 이름 허용 시 욕설 필터** 필요. 외형은 게놈에서 자동 생성 → 안전.

### 6.3 프레이밍 (슬롯 분리)
- 플레이어 생물을 **최종 보스로 쓰지 말 것** — "대멸종 피날레"와 "마지막" 슬롯이 충돌.
- **대멸종 = 환경 적합도 필터 피날레** / **플레이어 생물 = 그 앞의 "정점 라이벌(현 챔피언 격파)" 게이트.** 역할 분리로 둘 다 살린다.
- 더 안전한 첫 버전 대안: 플레이어 생물을 보스 슬롯이 아니라 **런 중 만나는 일반 야생 경쟁종**으로. 더 스포어답고 정밀 튜닝 부담이 적다.

### 6.4 지금 챙길 단 하나
- **게놈을 직렬화 가능 + 버전 붙은 구조로 설계** (§3.1). 그 외 멀티플레이어 백엔드는 전부 추후.

## 7. 가독성 & 연출 — 관전 게임의 생명줄

> 가만히 보는 게 지루하거나, "내 선택 → 이 결과"가 화면에서 안 읽히면 그냥 랜덤처럼 느껴져 게임이 죽는다. 이 레이어가 사실상 전부다.

- 왜 내 종이 이겼/졌는지가 **화면에서 읽혀야** 한다.
- 필요 요소: 실시간 인구수, 사망 원인 표시, 하이라이트(주목할 사건 자동 포착), 카메라 추적.
- 보스 위협 예고 UI (§4.2).

## 8. 기술 스택 & 작업 환경

### 8.1 스택
- **언어: TypeScript** (EOY · Be the Bee에서 쓰던 것과 동일 → 새 베팅 아님. 버전 붙은 게놈 같은 데이터 구조에 타입이 큰 도움).
- **렌더러: PixiJS** (WebGL 기반, 생물 수백~수천+ 마리 처리. 순수 Canvas는 보통 수백 마리에서 버벅임. Pixi는 "그리기"만 담당 → 시뮬 로직이 깨끗한 TS로 분리되어 Claude Code 생성에도 유리).
  - 대안: 입력·씬·에셋 로딩 내장이 필요하면 **Phaser**, 개체수가 적게 유지되면 **Canvas 2D**로도 충분.

### 8.2 모바일 대응 (접근을 뒤집는다)
- **모바일·세로 우선 설계 + 고정 논리 해상도 + scale-to-fit.**
- 폰에서 꽉 차고, 데스크톱에선 좌우 레터박스만 생긴다 → 비율 문제 소멸.
- 상호작용이 "카드 3장 중 탭 → 관전"이라 엄지 하나로 끝나는 저밀도 입력 → 폰이 오히려 이상적 디바이스.

### 8.3 배포 & 파이프라인
- 정적 호스팅(Vercel / Netlify / GitHub Pages) → **URL 하나.**
- 루프: **마크다운 설계 → Claude Code 구현 → 폰에서 URL 열어 검토.** 개발 루프 어디에도 데스크톱 불필요.
- **고도(Godot)는 현재 탈락** — 웹 익스포트를 만들려면 데스크톱 에디터가 필요. 데스크톱 가능해지면 이주 후보로 재검토.

## 9. 개발 단계 (Phased Roadmap)

> 각 단계 사이에 명시적 리뷰 체크포인트. 품질 우선, 단계별 검증 후 진행.

| Phase | 내용 | 검증 기준 |
|---|---|---|
| **0** | 기획 확정 (이 문서) | 코어 결정 합의 + §11 미해결 항목 정리 |
| **1** | 시뮬 코어 | 한 종이 정적 맵에서 채집/허기/죽음까지 그럴듯하게 |
| **2** | 게놈/형질 | 게놈을 바꾸면 생존 결과가 **눈에 띄게** 달라지는지 |
| **3** | 절차 환경 | 판마다 다른 맵 + 환경마다 유리 게놈이 달라지는지 |
| **4** | 카드 드래프트 메타루프 | 런 구조 완성 (드래프트→관전→정산 루프) |
| **5** | 보스 게이트 + 대멸종 피날레 | 위협 예고 + 보스별 위협 축 + 멸종 필터 |
| **6** | 가독성 / 연출 | 인구수·사망 원인·하이라이트·카메라 → "왜 이겼나"가 읽힘 |
| **S1** | (스트레치) 메타 언락 | 수평 확장, "한 판이 재밌다" 확인 후 |
| **S2** | (스트레치) 비동기 생물 | 수동 큐레이션 → 추후 서버 자동화 |

> **참고:** Phase 5 전까지 코어 싱글플레이 루프가 다른 플레이어 0명으로도 재밌어야 한다. 그게 먼저다.

## 11. 미해결 결정 — 다음 체크포인트에서 확정

> 구현 시작 전에 못 박아야 할 항목들. 일부는 기본값을 제안해 두었으나 확인 필요.

1. **화면 방향 (orientation)** — *제안: 세로(portrait) 기본.* 폰 검토 우선이라 엄지 친화적이지만, 생태계 시뮬의 "넓은 사바나 부감" 느낌은 가로가 유리. 레이아웃 전체를 좌우하는 결정 → **확인 필요.**
2. **승리 조건 수치** — 보스 개수, 런 길이(분), 라운드 수. (Everything is Crab 기준 런 ≈ 20분 / 보스 3마리가 참고점)
3. **플레이어 최적화 목표 정밀 정의** — "정점 포식자"를 기계적으로 무엇으로 측정할지 (천적 멸종? 정점 유지 N틱? 보스 통과만으로 충분?)
4. **플레이어 생물 프레이밍** — 정점 라이벌 게이트 vs 런 중 야생 경쟁종 (§6.3 중 택1)
5. **형질/카드 구체 목록** — Phase 2 진입 전 별도 패스로 작성
6. **타이틀 확정** — 현재 코드명 "적자생존" (친구 제안 키워드)
